# Muse Scoring Platform — 系统设计文档

> 本文档是 Muse 图片评分平台的权威架构说明书，描述系统的整体架构、模块设计、数据流、安全机制和运维要点。

---

## 一、系统概述

**项目定位**：面向公众的图片双维度评分平台，用户从"美学表现"和"细节完成度"两个维度（各 1-10 分）对图片进行标注。

**设计目标**：
- 支持百万级图片数据集的众包标注
- 高并发写入场景下的数据一致性与可靠性
- 友好的用户体验（温暖可爱的 UI 风格）
- 完善的管理后台与数据导出能力

### 5.2 表结构详述

#### users — 用户账户

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGSERIAL | PK | 自增主键 |
| username | VARCHAR(64) | UNIQUE NOT NULL | 登录名，正则 `[a-zA-Z0-9_]{3,20}` |
| password_hash | VARCHAR(128) | NOT NULL | bcrypt 哈希 |
| role | VARCHAR(16) | DEFAULT 'user' | `user` / `admin` |
| failed_attempts | INTEGER | DEFAULT 0 | 连续失败计数 |
| locked_until | TIMESTAMPTZ | NULLABLE | 锁定截止时间 |
| last_login | TIMESTAMPTZ | NULLABLE | 最后登录时间 |
| banned | BOOLEAN | DEFAULT FALSE | 永久封禁标记 |
| created_at | TIMESTAMPTZ | DEFAULT NOW() | 注册时间 |

**安全机制**：
- 连续 5 次密码错误 → 锁定 15 分钟
- 锁定时 `locked_until` 写入 UTC 时间戳
- 登录成功 → 重置 `failed_attempts = 0`，清除 `locked_until`
- `banned = TRUE` → 拒绝一切登录尝试

#### images — 图片元数据

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGSERIAL | PK | 自增主键（与图片文件名对应） |
| r2_url | TEXT | NOT NULL | Cloudflare R2 公开访问 URL |
| score_count | INTEGER | DEFAULT 0 | 已被多少人打过分 |
| deleted | BOOLEAN | DEFAULT FALSE | 软删除标记 |
| created_at | TIMESTAMPTZ | DEFAULT NOW() | 导入时间 |

**图片存储架构**：

```
开发环境（本地）:                    生产环境（Cloudflare R2）:

images/ ──┐                         images/ ──┐
          │ 读取文件名                          │ 读取文件
          ▼                                     ▼
   生成本地 URL                          boto3 上传到 R2
   http://localhost:8080/                https://images.yourdomain.com/
   images/001.webp                       001.webp
          │                                     │
          ▼                                     ▼
   存入 images.r2_url                    存入 images.r2_url
```

**为什么不用数据库存图片？**
- PostgreSQL 适合存结构化数据（URL、计数、元数据）
- 百万级图片二进制存数据库会导致：
  - 数据库体积暴增（TB 级别）
  - 备份/恢复极其缓慢
  - 查询性能严重下降
- R2 优势：
  - 零出口流量费（Cloudflare 免费额度充足）
  - 自带全球 CDN 加速
  - S3 兼容 API，boto3 直接调用
  - 无限存储扩展性

#### scores — 评分记录

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGSERIAL | PK | 自增主键 |
| image_id | BIGINT | FK → images.id | 被评分的图片 |
| user_id | BIGINT | FK → users.id | 评分的用户 |
| aesthetic_score | INTEGER | CHECK 1-10 | 美学维度分数 |
| completeness_score | INTEGER | CHECK 1-10 | 完成度维度分数 |
| submitted_at | TIMESTAMPTZ | DEFAULT NOW() | 提交时间 |

**关键约束**：
- `UNIQUE INDEX (user_id, image_id)` — 同一用户对同一图片只能打分一次
- `CHECK (aesthetic_score BETWEEN 1 AND 10)` — 数据库层面的分数范围校验
- `CHECK (completeness_score BETWEEN 1 AND 10)` — 同上

#### audit_exports — 审计日志

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGSERIAL | PK | 自增主键 |
| admin_id | BIGINT | FK → users.id | 执行导出的管理员 |
| export_type | VARCHAR(16) | | 导出格式（csv/jsonl） |
| filters | JSONB | | 导出时的过滤条件 |
| record_count | INTEGER | | 导出记录数 |
| exported_at | TIMESTAMPTZ | DEFAULT NOW() | 导出时间 |

### 5.3 索引策略

| 索引 | 列 | 类型 | 用途 |
|------|-----|------|------|
| `idx_scores_unique` | `(user_id, image_id)` | UNIQUE | 防止重复打分（ON CONFLICT 基础） |
| `idx_scores_user` | `(user_id)` | BTREE | 按用户查询评分记录 |
| `idx_scores_submitted` | `(submitted_at DESC)` | BTREE | 按时间倒序查询（导出/统计） |
| `idx_images_score_count` | `(score_count ASC)` | PARTIAL WHERE deleted=FALSE | 快速获取未标注图片 |

## 六、核心数据流

### 6.1 用户认证流程

```
用户提交登录/注册表单
        │
        ▼
  ┌─────────────┐
  │  前端校验    │  用户名格式(3-20位字母/数字/下划线)、密码长度(≥8位)、弱密码检测
  └──────┬──────┘
         │ 通过
         ▼
  ┌─────────────┐
  │ POST /api/  │
  │ auth/login  │
  │ 或 /register│
  └──────┬──────┘
         │
         ▼
  ┌─────────────────────────────────────────────┐
  │  后端处理                                     │
  │                                              │
  │ 注册: 检查用户名唯一 → bcrypt哈希 → INSERT   │
  │ 登录: 查用户 → 检查banned → 检查lockout      │
  │       → 验证密码 → 更新last_login/reset fail │
  │                                              │
  │ 返回: { token: JWT }                         │
  └──────┬───────────────────────────────────────┘
         │
         ▼
  ┌─────────────────────────────┐
  │  前端存储                    │
  │  localStorage.setItem()     │
  │  JWT payload 解析 role      │
  │  根据 role 跳转 admin/welcome│
  └─────────────────────────────┘
```

**JWT Token 结构**：
```json
{
  "sub": "用户ID (string)",
  "username": "用户名",
  "role": "user | admin",
  "exp": "过期时间戳 (7天后)"
}
```

### 6.2 打分流程（Write-Behind 架构）

```
用户点击"开始本轮打分"
        │
        ▼
┌───────────────────┐
│ Turnstile 验证     │  Managed 模式，开发环境可跳过
└────────┬──────────┘
         │ 通过
         ▼
┌───────────────────┐
│ POST /api/images/ │
│ batch             │  查询 score_count=0 的随机 10 张图片
└────────┬──────────┘
         │ 返回 [{id, r2_url}, ...]
         ▼
┌─────────────────────────────────────────────┐
│ 前端接收图片列表                              │
│ 从 r2_url 预加载图片 (new Image().src)       │
│ 图片直接从 Cloudflare R2 CDN 加载            │
│ 进入打分幻灯片 → 垂直 translate 切换          │
└────────┬─────────────────────────────────────┘
         │
         ▼ 用户选择两个维度的分数
┌───────────────────┐
│ POST /api/score   │
│ (携带 JWT)        │
└────────┬──────────┘
         │
         ▼
┌─────────────────────────────────────────────────────┐
│  FastAPI 处理（立即返回 200，不写 DB）                │
│                                                      │
│  1. Redis SADD user:{uid}:scored {image_id}         │
│     → 原子性去重，防止并发重复提交                    │
│  2. Redis EXPIRE scored_key 7天                      │
│  3. Redis SETEX dedup:submit:{uid}:{iid} 10s        │
│     → 短时间内的重复提交去重                          │
│  4. Redis LPUSH score_queue {json_payload}          │
│     → 写入异步队列，立即返回                          │
└────────┬────────────────────────────────────────────┘
         │
         ▼ 0.6s 后自动切换到下一张
┌───────────────────┐
│ 10 张全部打完     │
│ 进入结算页         │
└───────────────────┘

═══════════════════════════════════════════════════════
              后台 Worker 异步消费（独立协程）
═══════════════════════════════════════════════════════

┌─────────────────────────────────────────────────────┐
│  Redis Worker (main.py startup 启动)                │
│                                                      │
│  轮询间隔: 2 秒                                      │
│  批量大小: 1000 条                                   │
│                                                      │
│  1. LRANGE score_queue 0 999                        │
│  2. LTRIM score_queue 1000 -1  (原子弹出)           │
│  3. 组装批量 INSERT SQL (VALUES (...), (...), ...)  │
│  4. ON CONFLICT (user_id, image_id) DO NOTHING       │
│     → 保证幂等性，防止重试导致重复                    │
│  5. 更新 images.score_count (DISTINCT 防重复计数)   │
│  6. 同步 Redis Set user:{uid}:scored                │
│                                                      │
│  异常处理: 失败时将数据 LPUSH 回队列，等 5 秒重试    │
└────────┬────────────────────────────────────────────┘
         │
         ▼
┌───────────────────┐
│ PostgreSQL 批量写入│
│ ACID 事务保证      │
└───────────────────┘
```

| 函数 | 位置 | 说明 |
|------|------|------|
| `validate_username()` | auth.py | 正则校验 `[a-zA-Z0-9_]{3,20}` |
| `validate_password()` | auth.py | 长度≥8 + 弱密码黑名单 |
| `hash_password()` | auth.py | bcrypt 哈希 |
| `verify_password()` | auth.py | bcrypt 验证 |
| `create_token()` | auth.py | JWT 签发（7天有效期） |
| `get_current_user()` | auth.py | FastAPI 依赖，从 Token 提取用户 |
| `require_admin()` | auth.py | FastAPI 依赖，验证 admin 角色 |
| `check_account_lockout()` | auth.py | 检查锁定状态 |

**安全链**：
```
前端输入 → 格式校验 → 后端接收 → 用户名/密码校验
  → 查询用户 → 检查banned → 检查lockout → 验证密码
  → 成功: reset fail + update last_login → 返回JWT
  → 失败: increment fail → ≥5次则locked_until=now+15min
```

### 7.2 打分模块（routes.py）

**职责**：图片批次获取、评分提交、用户个人记录

**关键端点**：
| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/images/batch` | POST | 获取 10 张未标注图片（需 Turnstile） |
| `/api/score` | POST | 提交评分到 Redis 队列 |
| `/api/my-scores` | GET | 获取当前用户已打分的图片 ID 列表 |
| `/api/my-stats` | GET | 获取用户个人统计（总分/今日分） |

**去重机制（双重保障）**：
1. **Redis Set** `user:{uid}:scored` — 原子性 `SADD` 判断是否已打分
2. **PostgreSQL UNIQUE INDEX** `(user_id, image_id)` — 数据库层面的最终防线
3. **Redis SETEX** `dedup:submit:{uid}:{iid}` — 10 秒窗口内的重复提交防护

### 7.3 Redis Worker（redis_worker.py）

**职责**：后台异步消费 Redis 队列，批量写入 PostgreSQL

**运行机制**：
- 在 `main.py` 的 `startup` 事件中通过 `asyncio.create_task()` 启动
- 作为 FastAPI 事件循环中的后台协程持续运行
- 不需要独立的进程或容器

**批量写入策略**：
```python
# 动态构建多值 INSERT
INSERT INTO scores (...) VALUES
  (:v0_image_id, :v0_user_id, ...),
  (:v1_image_id, :v1_user_id, ...),
  ...
  (:v999_image_id, :v999_user_id, ...)
ON CONFLICT (user_id, image_id) DO NOTHING
RETURNING image_id
```

**容错机制**：
- 写入失败 → `LPUSH` 回队列 → 等待 5 秒 → 重试
- `ON CONFLICT DO NOTHING` → 幂等性保证
- `DISTINCT` → 防止重试时重复计数

## 八、安全设计

### 8.1 认证安全

| 措施 | 实现 | 防御目标 |
|------|------|----------|
| bcrypt 密码哈希 | passlib[bcrypt] | 密码泄露后无法逆向 |
| JWT 7 天过期 | python-jose exp claim | Token 泄露后自动失效 |
| 账户锁定机制 | 5 次失败锁定 15 分钟 | 暴力破解防护 |
| 弱密码黑名单 | 内置常见弱密码集合 | 弱密码防护 |
| 用户名格式限制 | 正则 `^[a-zA-Z0-9_]{3,20}$` | SQL 注入/XSS 预防 |
| 角色隔离 | `require_admin()` 依赖 | 越权访问防护 |
| 封禁机制 | `banned` 字段 | 恶意用户封禁 |

### 8.2 数据安全

| 措施 | 实现 | 防御目标 |
|------|------|----------|
| UNIQUE INDEX | `(user_id, image_id)` | 重复打分防护 |
| CHECK 约束 | 分数 1-10 范围 | 非法数据入库防护 |
| ON CONFLICT DO NOTHING | Worker 批量插入 | 幂等性保证 |
| 参数化查询 | SQLAlchemy text() + 命名参数 | SQL 注入防护 |
| 审计日志 | audit_exports 表 | 导出操作可追溯 |

### 8.3 网络安全

| 措施 | 实现 | 防御目标 |
|------|------|----------|
| CORS 配置 | 限定 origins/methods/headers | 跨域攻击防护 |
| Nginx 安全头 | X-Frame-Options, X-Content-Type-Options, X-XSS-Protection | 点击劫持/MIME/XSS 防护 |
| Turnstile 验证 | Cloudflare 人机验证 | 自动化刷分防护 |
| 全局异常处理 | 统一 500 响应，不泄露堆栈 | 信息泄露防护 |

## 十、性能设计

### 10.1 Write-Behind 缓冲

打分提交采用异步架构：
- 用户打分 → Redis LPUSH → **立即返回**（毫秒级响应）
- Worker 每 2 秒或攒满 1000 条 → 批量 INSERT → 持久化

这确保了在高并发场景下，用户感受到的延迟仅为网络 RTT + Redis 写入（通常 < 10ms），而非数据库写入。

### 10.2 连接池

- **PgBouncer**：transaction 模式，最大 500 客户端连接，池大小 25
- **SQLAlchemy**：pool_size=20, max_overflow=10
- 双层连接池确保数据库不会被突发流量击垮

### 10.3 前端优化

- **图片预加载**：获取 batch 后立即预加载全部 10 张图片
- **Gzip 压缩**：Nginx 对 JS/CSS/JSON 启用 Gzip
- **静态缓存**：JS/CSS/图片设置 1 年缓存 + immutable
- **SPA 路由**：try_files 回退到 index.html

### 10.4 数据库索引

- `idx_images_score_count` 为部分索引（`WHERE deleted = FALSE`），精准定位未标注图片
- `idx_scores_submitted` 降序索引，优化时间范围查询和最新记录检索

## 十一、扩展性考虑

### 11.1 水平扩展

当前架构支持以下扩展方向：

| 组件 | 扩展方式 | 说明 |
|------|----------|------|

- **API 版本控制**：路由前缀 `/api/` 为后续 `/api/v2/` 预留空间
- **WebSocket 支持**：Nginx 已配置 Upgrade/Connection 头，支持实时通知
